Every session starts cold

Three things that slow down every developer who uses Claude Code:

  • 🧊 Cold start, every time. You explain the same architecture, the same conventions, the same deployment steps — every single session. Claude has no memory.
  • 📋 Lost decisions. You decided to use Drizzle instead of Prisma for a good reason. Three weeks later, Claude is suggesting Prisma again. The context is gone.
  • 🔁 Repeated context. Every new chat: "This is a Next.js 15 project, we use Tailwind, ESLint config is at .eslintrc.json..." The same 200 words, every time.

Install in 30 seconds

Add to your Claude Code config. No account, no API key, no cloud.

~/.claude.json

{
  "mcpServers": {
    "memory": {
      "command": "npx",
      "args": ["-y", "@gilligan-tech/claude-memory"]
    }
  }
}

Restart Claude Code. Done. Your memory database lives at ~/.claude-memory/memory.db — local SQLite, never leaves your machine.

Or install globally: npm install -g @gilligan-tech/claude-memory then use "command": "claude-memory"

Ten tools

Claude Code calls them automatically — or you ask it to.

Tool What it does
memory_bootstrap Load all memories for a project at session start. Call this first.
memory_save Persist a note, decision, rule, or architecture fact.
memory_recall Search by keyword — FTS5/BM25 ranking, optional tag filter, recency-weighted results.
memory_update Correct an existing memory when things change.
memory_delete Permanently remove a memory by ID.
memory_archive Soft-hide a memory (reversible) — excluded from recall and bootstrap by default.
memory_stats Count breakdown by project and type — e.g. "42 memories · 6 projects".
memory_list Paginated listing of all memories without a search query.
memory_export Export memories to JSON — back up or move between machines.
memory_import Import from JSON in merge or replace mode.

Use type to categorize: rules, architecture, deploy, decision, preference, note. Tag memories with tags for AND-filtered recall. Scope with repo, or leave empty for globals that surface in every project.

How it works

Dead simple. No vectors, no embeddings, no cloud calls.

Session A
You save

Tell Claude something worth keeping. It calls memory_save.

→
Storage
Local SQLite

Stored in ~/.claude-memory/memory.db on your machine.

→
Session B
Claude loads

Next session calls memory_bootstrap. All context restored.

Search uses SQLite FTS5 (full-text search with BM25 ranking and a light recency boost) — fast, offline, no API calls. No OpenAI embeddings. No Qdrant. Everything runs on your laptop.
