Install in 30 seconds

Add one block to your config. No account, no API key, no cloud.

~/.claude.json
{
  "mcpServers": {
    "memory": {
      "command": "npx",
      "args": ["-y", "@gilligan-tech/claude-memory"]
    }
  }
}
claude_desktop_config.json
{
  "mcpServers": {
    "memory": {
      "command": "npx",
      "args": ["-y", "@gilligan-tech/claude-memory"]
    }
  }
}

Restart Claude. Done. Your memory database lives at ~/.claude-memory/memory.db — local SQLite, never leaves your machine.

Faster startup? Install globally: npm install -g @gilligan-tech/claude-memory then use "command": "claude-memory".

Six types of memory

Tag each note so Claude knows what matters most — rules and architecture surface first at session start.

rules
Coding conventions, style preferences, hard constraints.
e.g. "Always use async/await, never callbacks."
architecture
System design, component relationships, patterns.
e.g. "Routes → services → repository. No logic in routes."
deploy
Deployment steps, server configs, environment notes.
e.g. "./deploy.sh to prod. Requires VPN."
decision
Why you chose X over Y — with the rationale.
e.g. "Drizzle over Prisma — migrations too slow."
preference
Personal working preferences across projects.
e.g. "Terse commit messages, no emoji."
note
Everything else worth keeping around.
e.g. "Staging DB resets every Sunday at 02:00 UTC."

Ten tools

Claude Code calls them automatically — or you ask it to.

ToolWhat it does
memory_bootstrapLoad all memories for a project at session start. Call this first.
memory_savePersist a note, decision, rule, or architecture fact.
memory_recallSearch by keyword — FTS5/BM25 ranking, optional tag filter, recency-weighted.
memory_updateCorrect an existing memory when things change.
memory_deletePermanently remove a memory by ID.
memory_archiveSoft-hide a memory (reversible) — excluded from recall and bootstrap by default.
memory_statsCount breakdown by project and type — e.g. "42 memories · 6 projects".
memory_listPaginated listing of all memories without a search query.
memory_exportExport memories to JSON — back up or move between machines.
memory_importImport from JSON in merge or replace mode.

Tag memories for AND-filtered recall. Scope with repo, or leave it empty for globals that surface in every project.

No cloud. No account. No telemetry.

Your memory database is a single plain SQLite file at ~/.claude-memory/memory.db. Inspect it with any SQLite client, back it up with memory_export, and move it between machines whenever you like. Nothing is sent anywhere — there's no server to send it to.

Questions

The things developers ask before installing.

Does it work offline?
Completely. Everything is local SQLite — there are no network calls, no API keys, and no cloud service. It runs the same on a plane as it does at your desk.
Can I use it on multiple machines?
Yes. Use memory_export to dump your memories to JSON and memory_import to load them elsewhere. Or just sync the ~/.claude-memory/memory.db file via Dropbox, iCloud, or git.
Will it slow down Claude Code?
No. Reads are synchronous SQLite queries backed by an FTS5 index — sub-millisecond on a normal memory store. There's no network round-trip to wait on.
How do I see what's saved?
Ask Claude to run memory_list or memory_stats, or open ~/.claude-memory/memory.db in any SQLite viewer. It's a plain, inspectable database — nothing proprietary.
Does it work with Claude Desktop too?
Yes. It's a standard MCP server, so the same config works in Claude Desktop, Claude Code, and any other MCP-compatible host. Just add the block to that app's config file.
What does it depend on?
Node.js 18+ and better-sqlite3, which ships prebuilt binaries for every major platform — no compiler needed. That's it.
